# AI 客服系统产品页面设计 Review

> 版本：v8.0  
> 日期：2026-04-12  
> 范围：前端交互体验 / 管理后台设计 / 架构优化

---

## 一、总体评价

现有设计文档在**后端架构、认证安全、微服务拆分、云部署**方面已经非常完善（v2.1 Review 修订版修复了 35 个安全与一致性问题）。但在**产品页面设计（前端用户体验）**方面存在较大缺失——目前只定义了 SSE 事件流和少量组件列表，缺少对**实际 UI/UX 交互设计**的详细描述。

以下从产品页面设计的角度，提出 6 个维度的优化建议。

## 二、客服对话页面优化方案

### 2.1 当前设计的不足

| 维度 | 现状 | 问题 |
|------|------|------|
| 登录前页面 | 仅提到 OAuth 按钮 | 缺少品牌介绍、功能预览、未登录状态引导 |
| 对话体验 | SSE 文本/卡片 | 缺少打字效果、流式渲染、消息气泡、错误重试 |
| 多轮对话 | 四层内存 | 用户侧无对话历史列表 |
| 快捷功能 | 快捷菜单 JSON | 缺少场景化引导、智能推荐时机 |
| 异常处理 | 错误码 | 缺少网络异常、Token 过期、服务不可用的降级体验 |
| 响应式 | 未提及 | 未适配移动端（微信小程序） |

### 2.3 关键交互细节

**1. 微信小程序静默登录引导页面**

当前设计缺少品牌展示和功能引导。建议新增：

```

│         [企业 LOGO]                   │
│                                      │
│    AI 智能客服 24 小时在线             │
│    订单查询 · 产品咨询 · 售后支持       │
│                                      │
│    ┌────────────────────────────┐   │
│    │  🟢 微信授权登录             │   │
│    └────────────────────────────┘   │
│                                      │
│    已有账号？手机号登录 →              │
│                                      │
│    [功能预览] [隐私政策] [联系我们]    │
└──────────────────────────────────────┘
```

> 微信小程序环境下使用 `wx.login` 静默授权，用户首次进入时通过微信授权按钮完成登录，无需额外跳转 OAuth 页面。

**2. 流式渲染优化**

SSE 流式输出时，前端不应等待完整消息再渲染，而应该：

```typescript
// 推荐方案：逐字渲染 + 打字机效果
const [streamingText, setStreamingText] = useState("");

// 收到 SSE text 事件
onSSEMessage((data) => {
  if (data.type === "text") {
    setStreamingText(prev => prev + data.content);  // 增量追加
  }
  if (data.type === "loading") {
    setShowLoadingIndicator(true);
  }
});

// 渲染
{streamingText && (
  <TypewriterText text={streamingText} speed={20} />
)}
```

**3. 对话历史侧边栏**

用户应能看到过往对话记录，支持：
- 自动按日期分组（今天/昨天/更早）
- 点击历史会话恢复上下文
- 一键清除当前会话

```

│  📋 历史对话                    [ + 新对话]│
│  ───────────────────────────────────     │
│  今天                                     │

│  └── 售后咨询 (09:15)                    │
│  昨天                                     │
│  ├── 产品推荐 - 遮光窗帘                  │
│  └── 物流问题                            │
│  更早                                     │
│  └── 退换货申请                          │
└─────────────────────────────────────────┘
```

**4. 智能推荐时机**

不应始终显示推荐，而应基于上下文动态触发：

| 触发场景 | 推荐内容 |
|---------|---------|
| 用户首次打开对话页 | 快捷功能入口（查订单、找产品、售后） |
| 查询订单完成后 | 查看物流、申请售后、查看历史订单 |
| 产品咨询完成后 | 查看同类商品、加入购物车、客服咨询 |
| 售后申请完成后 | 查看售后进度、取消申请、联系客服 |
| 用户连续发送短句（情绪判断） | 转接人工客服 |
| AI 连续 2 次低置信度 | 转接人工客服 + 问题总结 |

**5. 异常降级体验**

| 异常场景 | 用户体验 |
|---------|---------|
| 网络断开 | 显示"网络已断开"灰色提示，保留输入框，恢复后自动重发 |
| Token 过期 | 静默刷新，刷新失败则弹出登录引导页，不中断当前对话 |
| AI 服务不可用 | "AI 客服正在维护中，请稍候" + 直接转人工入口 |
| 百炼 API 超时 | "回答生成中，请稍等..." 超时后提供"重试"按钮 |
| 用户发送图片 | "目前暂不支持图片识别，请用文字描述您的需求" |

### 2.4 移动端适配要求

| 渠道 | 技术要求 |
|------|---------|
| **微信小程序** | 使用 wx.login 静默授权 + 小程序原生组件 |
| **浏览器** | 响应式布局（320px - 1920px）、PWA 支持 |

**关键差异**：
```
桌面端（>1024px）：
- 左右布局：左侧历史会话 + 右侧对话
- 宽卡片消息区可展示详细信息

移动端（<768px）：
- 全屏对话，历史会话通过侧滑抽屉
- 卡片消息简化，支持点击展开
- 输入框始终固定在底部安全区域
```

## 四、架构层面优化建议

### 4.1 前端技术选型优化

| 当前方案 | 优化建议 | 原因 |
|---------|---------|------|
| Next.js SSG (OSS 部署) | Next.js SSR 部署到 SAE | 支持 SSR 渲染、更好的 SEO、动态内容 |
| 纯 OSS 静态托管 | OSS + 函数计算（SSR） | 保持低成本，同时支持 SSR |
| 无状态管理 | 考虑使用 Vercel / Netlify 部署前端 | 更简单的前端部署流程 |

**推荐方案（已确认）**：
- 客服对话页：SPA（纯前端，OSS + CDN），无需 SSR
- 管理后台：SSR（Next.js 部署到 SAE），支持服务端权限校验和动态内容

### 4.2 数据库 Schema 补充

当前设计缺少以下关键表：

| 表名 | 用途 | 优先级 |
|------|------|--------|
| `conversations` | 对话会话（含用户消息和 AI 回复） | P0 |
| `conversation_messages` | 对话消息详情（支持历史记录查询） | P0 |
| `knowledge_documents` | 知识库文档（本地管理，同步到百炼） | P1 |
| `service_tickets` | 人工客服工单 | P1 |
| `tool_execution_logs` | Tool 执行日志（用于调试分析） | P1 |
| `rate_limit_config` | 租户级别限流配置 | P2 |

### 4.4 前端与后端的契约测试

建议在 API Reference 文档基础上，增加 OpenAPI/Swagger 定义：

```yaml
# openapi.yaml（自动生成前端类型定义）
openapi: "3.0.3"
info:
  title: AI 客服 API
  version: "2.1"
paths:
  /api/chat/messages:
    post:
      summary: 发送消息（SSE 流式）
      requestBody:
        content:
          application/json:
            schema:
              $ref: "#/components/schemas/ChatMessageRequest"
      responses:
        "200":
          description: SSE 事件流
```

这样可以使用 `openapi-typescript` 自动生成前端 TypeScript 类型，避免前后端接口不一致。

## 六、总结

**现有架构优势**：后端架构、认证安全、微服务拆分、云部署方案已经非常完善。

**需要补强的方向**：

1. **前端交互设计**需要详细定义（当前仅有 API 级别的 SSE 格式）
2. **管理后台**需要明确布局结构和 AI 助手的集成方式
3. **数据库 Schema**需要补充 conversation、knowledge_documents、tool_execution_logs 等关键表
4. **前端组件库**应独立为 packages 共享组件
5. **移动端适配**需要明确的渠道策略（微信小程序）

建议按照以下优先级推进：
- **Phase 5 前端对话界面**：先完成最小可用对话页（登录 → 对话 → SSE 流式回复）
- **Phase 6 AI 管理助手**：先完成管理首页 + AI 洞察面板
- **Phase 7 移动端适配**：微信小程序优先
- **Phase 8 体验优化**：对话历史、智能推荐、异常降级
